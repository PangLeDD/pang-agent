# Agent 架构设计

> 日期：2026-07-08
> 状态：讨论中，阶段性总结

## 1. 项目定位

独立于 Web 后台的 AI Agent 助手，用户通过自然语言操作 Web 系统。

- 后端：FastAPI
- Agent 框架：LangGraph（图编排）
- 数据库：PostgreSQL + SQLAlchemy（异步）
- 日志：Loguru
- 认证：RBAC（JWT + passlib）
- 包管理：uv

**目标**：生产级 Agent 铁三角 —— 省 Token、准（确定性）、快（低延迟）

---

## 2. 整体架构（6 层）

```
┌─────────────────────────────────────────────────────┐
│  入口层   │  FastAPI 接收请求，WS/SSE 流式输出        │
├─────────────────────────────────────────────────────┤
│  路由层   │  意图分类 → 参数提取 → 四层校验 → 追问    │
├─────────────────────────────────────────────────────┤
│  编排层   │  域 Agent 调度、状态机、上下文装配、       │
│ (Harness) │  Token 预算、记忆机制                     │
├─────────────────────────────────────────────────────┤
│  执行层   │  设备域│订单域│用户域│监控域 Agent         │
│           │  (每个只带本域 5-15 个 Tool 的 ReAct)      │
├─────────────────────────────────────────────────────┤
│  资源层   │  Tool 注册中心、Mock/真实 API 适配、DB、    │
│           │  Memory Store                            │
├─────────────────────────────────────────────────────┤
│  观测层   │  日志(Loguru)、指标收集、评测框架           │
└─────────────────────────────────────────────────────┘
```

### 2.1 为什么不用单一 ReAct Agent？

- 50+ Tool Schema 全塞 context → Token 爆炸
- 每次推理遍历所有工具 → 慢
- 工具多了大模型易选错 → 不准

### 2.2 路由 + 域 Agent 模式

```
用户输入 → 意图路由（三级漏斗）→ 域Agent(仅本域Tool) → 结果
              ↓ (兜底)
          通用 ReAct Agent
```

- 意图路由：关键词匹配 → embedding 相似度 → 小模型分类
- 每个域 Agent context 只含 5-15 个 Tool，Token 减少 60%-90%

## 3. 路由层：意图识别与追问

详见独立文档：[Agent 意图识别与追问澄清机制：生产级工程化完全指南](../Agent%20意图识别与追问澄清机制：生产级工程化完全指南.md)

核心原则：

| 流程环节            | 是否用 LLM | 实现方式                     |
| ------------------- | ---------- | ---------------------------- |
| 意图分类            | 不用       | 规则 / 小模型分类 / 向量检索 |
| 参数提取            | 必须用     | 专属 Prompt + Few-shot        |
| 完整性 & 格式校验   | 完全不用   | 代码硬校验                   |
| 追问话术生成        | 条件触发   | 高频模板，复杂场景用轻量 LLM  |
| 多轮状态管理        | 完全不用   | Redis / PG 存储状态机        |
| 最终执行            | 视复杂度   | 简单查询直接返回，复杂走 Agent |

**四层校验体系**：
1. 必填缺失校验
2. 格式/值域校验
3. 歧义/多候选冲突消解
4. 置信度兜底防护

> **核心认知**：永远不要让 LLM 自己判断"是否需要追问"。LLM 只负责"提取"和"生成追问话术"，"是否追问"永远由代码规则决策。

## 4. 编排层 (Harness)：驾驭 Agent 的缰绳

编排层模块清单：

1. 状态机（context state）
2. 上下文装配（Context Builder / Token 预算）
3. 记忆机制（Memory）
4. 域 Agent 调度

## 5. 记忆机制：五层记忆模型

```
Agent
    │
Memory Manager (抽取、去重、合并、过期)
    │
 ┌──┴──────────────────────────────┐
 │  Working    │  状态机维护，程序驱动   │ 确定性信息，零幻觉  │
 │  Memory     │  pending_params 等    │                    │
 │─────────────┼───────────────────────┼────────────────────│
 │  Convers.   │  最近 5 轮原文         │ 保留代词、省略等   │
 │  Memory     │  最多 ~3000 tokens     │ 不做摘要           │
 │─────────────┼───────────────────────┼────────────────────│
 │  Semantic   │  Entity-Relation-Fact  │ 用户事实、设备事实 │
 │  Memory     │  PG 建表，非 KV 单表   │ 公司知识           │
 │─────────────┼───────────────────────┼────────────────────│
 │  Episodic   │  事件记忆              │ 工具调用历史       │
 │  Memory     │  什么时候/发生了什么/结果│ 成功/失败/耗时     │
 │─────────────┼───────────────────────┼────────────────────│
 │  Procedural │  经验与工作流           │ 按 trigger 匹配   │
 │  Memory     │  自动提炼 Best Practice │ v2 阶段启用       │
 └─────────────┴───────────────────────┴────────────────────┘
```

### 5.1 各层详细定位

#### Working Memory（运行状态）

```
current_intent, pending_slots, tool_running, tool_result, current_plan, retry_count
```

- 程序维护，不让 LLM 自己总结
- 和追问状态机关联

#### Conversation Memory（最近聊天）

- 保留最近 5 轮原文，不做摘要
- 语言风格、代词、省略都靠这里维持

#### Semantic Memory（事实记忆）

核心结构：Entity + Relation + Fact（不是 KV 键值对）

```
Entity:  Device#9374, Building#A, User#Tom
Relation: Device9374 --located_in--> BuildingA
Fact:    temperature=36, status=offline
```

- 查询"设备在哪"直接查 Entity Relation，不等同于全文检索
- PG 建表，结构化存储

#### Episodic Memory（事件记忆）

记录：

```
时间 → 做了什么 → 结果
```

例如：

```
2025-05-01, 用户查询 9374, 工具调用失败(超时), 重新调用成功
```

- 用于评测回溯、错误分析、经验提炼
- v1 记录工具调用历史（成功/失败/耗时）

#### Procedural Memory（经验）

``` 
If query_device:
  → 获取 token → 调用 API → 解析 JSON → 渲染回复
  
If device offline:
  → 检查最近告警 → 检查网关状态 → 排除法定位
```

- 需要足够量级的 Episodic Memory 积累后自动提炼
- v2 启用，v1 先埋点

### 5.2 Memory Manager

独立模块，负责：

- 决定哪些信息值得长期保存（不是每句话都存）
- 合并重复事实，解决新旧冲突
- 设置 TTL（临时上下文自动过期）
- 标签化（importance、entity、intent）
- 版本与来源标记

### 5.3 Memory Retrieval：多信号融合打分

不只用向量检索，而是多信号加权：

```
Score = 0.4 × semantic_similarity
      + 0.3 × entity_match
      + 0.2 × recency
      + 0.1 × importance
```

- Recency：最近发生优先
- Semantic：与当前问题最相关
- Entity：涉及相同实体优先
- Importance：用户偏好、关键配置等高权重
- Intent：按意图过滤无关记忆

### 5.4 v1 实施优先级

| 层 | v1 必要性 | MVP 形态 |
|---|---|---|
| Working Memory | **必须** | 状态机驱动，和追问联动 |
| Conversation Memory | **必须** | 最近 5 轮原文 |
| Semantic Memory | **必须** | Entity-Relation-Fact，PG 建表 |
| Episodic Memory | **强烈建议** | 工具调用历史（成功/失败/耗时）|
| Procedural Memory | **v2** | 先埋点记录 |
| Memory Manager | **v1 骨架** | 提取事实 + 去重，评分简化版 |
